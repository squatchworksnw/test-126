# SquatchWorks Pilot Validation Script

Purpose: validate whether real people can use SquatchWorks for daily operations with minimal training.

This pilot is not a demo. Do not explain how every screen works first. Watch where people naturally go, where they hesitate, and where they ask for help.

## Pilot Roles

- Submitter: staff, volunteer, or office user who reports needs.
- Field Worker: person assigned to do work.
- Reviewer/Admin: person who reviews, assigns, and approves operational records.

## Facilitator Script

### Opening

Thank you for helping test SquatchWorks. We are testing the system, not you. If something is confusing, that is useful feedback.

Please say what you are thinking while you work. If you are unsure what to do, say that out loud before asking for help.

I may stay quiet for a bit so we can see what the screen communicates on its own. If you get stuck, I will help after noting where it happened.

### Rules

- Use the app as naturally as possible.
- Do not worry about making mistakes.
- Ask for help whenever you would in real life.
- The observer should not coach unless the tester is stuck for more than 60 seconds.

### After Each Task

Ask:

- Was that easy, okay, or hard?
- What word or screen confused you?
- What did you expect to happen next?
- Would you feel comfortable doing this alone next week?

## Things To Watch For

### Confusing Screens

- Screens with too many equal choices.
- Screens where the primary action is not obvious.
- Screens where the user reads for more than 10 seconds before acting.
- Screens where the user says, "I don't know where to go."

### Unclear Labels

- Words users do not repeat back correctly.
- Labels that sound similar, such as Work, Work Orders, My Work, Assigned Work, Scheduled Work.
- Labels that feel technical or administrative.
- Status words that do not tell the user what to do next.

### Places Users Stop

- More than 10 seconds without action.
- Cursor or finger moves between multiple choices.
- User opens the wrong screen.
- User backs out and tries another path.
- User says, "Wait," "Where is," "Do I," or "What does this mean?"

### Places Users Ask For Help

Record the exact question. The question is often the product requirement.

Examples:

- "Where do I send this?"
- "Did it save?"
- "Is this assigned to me?"
- "Where did the photo go?"
- "How do I know this is done?"
- "Where is the old repair?"

## Success Metrics

### Task Success

- Pass: user completes task without help.
- Assisted pass: user completes task after one hint.
- Fail: user cannot complete task or needs step-by-step help.

Target for pilot readiness:

- Submit request: 90% pass or assisted pass.
- Upload photo: 80% pass or assisted pass.
- Find assigned work: 85% pass or assisted pass.
- Complete task: 85% pass or assisted pass.
- Find vehicle repair history: 75% pass or assisted pass.
- Locate warranty: 75% pass or assisted pass.
- Admin review approval: 80% pass or assisted pass.

### Time To Confidence

Measure from task start until user says or demonstrates they know what to do.

Targets:

- Submit request: under 2 minutes.
- Check request status: under 1 minute.
- Upload photo: under 2 minutes.
- Find assigned work: under 45 seconds.
- Complete task: under 1 minute.
- Find vehicle repair history: under 2 minutes.
- Locate warranty: under 2 minutes.
- Review and approve request: under 3 minutes.

### Help Requests

Track:

- Number of help requests per task.
- Type of help requested.
- Whether help was about navigation, wording, confidence, or missing data.

Target:

- No task should average more than 1 help request.

### Confidence Rating

After each task, ask:

How comfortable would you feel doing this alone?

Score:

- 1: I could not do this alone.
- 2: I would need help.
- 3: I might need help.
- 4: I could probably do it.
- 5: I could do this confidently.

Target:

- Average 4 or higher for core daily tasks.
- No role-critical task below 3.

## Severity Guide

P0: Blocks task completion or creates risk of lost operational data.

P1: User can finish, but confusion or help is likely.

P2: Polish, wording, layout, or training improvement.

## Recommended Pilot Size

Run at least:

- 2 submitters.
- 2 field workers.
- 1 reviewer/admin.
- 1 older or low-technical-confidence user if available.

Run sessions individually. Group testing hides hesitation.
